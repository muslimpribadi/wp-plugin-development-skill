# Block Editor Fundamentals

## 1. Project Structure & Build Flow

### Core Constraint
Blocks must be registered within **plugins**, not themes. This ensures portability and separation of concerns. Adherence to the `create-block` tool structure is recommended but not mandatory.

### Directory Architecture

| Path | Purpose |
| :--- | :--- |
| `src/` | Source code (ESNext, JSX). Written by developer. |
| `build/` | Compiled output (JS/CSS). Generated by build tools. |
| `package.json` | Dependencies and build scripts. |
| `composer.json` | PHP dependencies. |
| `plugin.php` | Main plugin file (entry point). |

### Source (`src/`) & Build (`build/`) Relationship
The build process transforms files from `src/` into `build/`. The plugin loads assets and logic from the `build/` directory.

- **Written to:** `src/` (or root for PHP).
- **Loaded by WP:** `build/`.

### File Inventory & Roles

| File | Context | Description |
| :--- | :--- | :--- |
| `block.json` | Metadata | Defines block name, title, attributes, supports, and paths. |
| `edit.js` / `index.js` | Client (Editor) | React component for the block editor interface. |
| `save.js` | Client (Save) | React component returning static HTML/JSX for the database. |
| `render.php` | Server (Dynamic) | PHP callback for dynamic front-end rendering. |
| `style.css` / `editor.css` | Styles | Front-end and editor-specific styling. |
| `view.js` | Client (View) | Script enqueued only on the front end for interactivity. |
| `view.php` | Server (View) | PHP logic executed only on the front end. |

### Build Process Constraints

1.  **Initialization**: Run `npm init @wordpress/block my-plugin` in the plugin root directory.
2.  **Development Loop**: Execute `npm start`. Watch mode remains active, compiling changes from `src/` to `build/` automatically.
3.  **Production Build**: Execute `npm run build`. Compiles and minifies assets for deployment.
4.  **Execution Context**: The plugin must be installed and activated in the WordPress installation where development occurs.

### Asset Loading Rules
-   Scripts: Defined in `block.json` using keys like `editorScript`, `script`, or `viewScript`.
-   Styles: Defined in `block.json` using keys like `style`, `editorStyle`, or `viewStyle`.
-   The build tool automatically resolves these paths and enqueues them correctly.

## 2. `block.json` Schema & Configuration

### Core Purpose
`block.json` is the single source of truth for block registration. It defines metadata, asset paths, attributes, and supports for both client (JS) and server (PHP) contexts.

### Canonical JSON Schema

```json
{
  "$schema": "https://schemas.wp.org/trunk/block.json",
  "apiVersion": 3,
  "name": "namespace/block-name",
  "version": "0.1.0",
  "title": "Block Title",
  "category": "design",
  "icon": "smiley",
  "description": "Block description.",
  "textdomain": "text-domain",
  "editorScript": "file:./build/index.js",
  "editorStyle": "file:./build/index.css",
  "style": "file:./build/style.css",
  "script": "file:./build/script.js",
  "viewScript": "file:./build/view.js",
  "render": "file:./render.php",
  "attributes": {
    "key": { "type": "string", "default": "" }
  },
  "supports": {
    "color": { "text": true, "background": true }
  }
}
```

### Property Groups

#### 1. Basic Metadata
Defines identity and editor UI placement.

| Property | Type | Description |
| :--- | :--- | :--- |
| `name` | string | **Required.** Unique identifier (e.g., `my-plugin/my-block`). Must match directory name. |
| `title` | string | Display title in the Inserter. |
| `category` | string | Inserter category (`text`, `media`, `design`, `widgets`, `theme`, `reusable`). |
| `icon` | string | Dashicon slug or SVG markup. |
| `description` | string | Contextual description shown in the Inserter. |
| `keywords` | array | Search keywords for the Inserter. |
| `textdomain` | string | Text domain for internationalization (`__()`). |
| `apiVersion` | integer | Block API version (default: `3`). |

#### 2. Asset Registration
Paths to JS/CSS files. Resolved relative to `block.json`. Supports `file:` prefix or registered handles.

| Property | Context | Description |
| :--- | :--- | :--- |
| `editorScript` | Editor Only | Main React entry point for the block editor. |
| `script` | Editor + Frontend | Shared JavaScript logic. |
| `style` | Editor + Frontend | Shared CSS styles. |
| `editorStyle` | Editor Only | Styles specific to the block editor UI. |
| `viewScript` | Frontend Only | Interactivity scripts loaded only on the front end. |
| `viewStyle` | Frontend Only | Styles loaded only on the front end. |

#### 3. Rendering & Output
Controls how the block is rendered and saved.

| Property | Type/Context | Description |
| :--- | :--- | :--- |
| `render` | string (Path) | Path to PHP file for dynamic rendering (WP 6.1+). |
| `save` | **Deprecated** | Use `save.js` in conjunction with `view.php` or static save logic instead. |

#### 4. Attributes (`attributes`)
Defines custom data stored in the database. Parsed from block delimiters.

```json
"attributes": {
    "content": {
        "type": "string",
        "source": "html",
        "selector": "p",
        "default": ""
    },
    "align": {
        "type": "string",
        "enum": ["left", "center", "right"],
        "default": "left"
    }
}
```

**Key Types:** `string`, `boolean`, `number`, `array`, `object`.
**Storage:** Serialized into the block comment delimiter by default.

#### 5. Supports (`supports`)
Enables UI controls in the Block Editor Settings Sidebar.

```json
"supports": {
    "align": ["wide", "full"],
    "color": {
        "text": true,
        "link": true,
        "background": true,
        "gradients": true
    },
    "spacing": {
        "padding": true,
        "margin": true
    },
    "typography": {
        "fontSize": true,
        "lineHeight": true
    }
}
```

**Note:** Enabling supports generates CSS classes/inline styles that must be applied to the wrapper element in `save.js` or `render.php`.

### Block Delimiters & Storage
Blocks are stored in post content using HTML comment delimiters:

```html
<!-- wp:namespace/block-name {"align":"full","style":{"spacing":{"padding":{"top":"16px"}}}} -->
<div class="wp-block-namespace-block-name has-align-full" style="padding-top:16px">
    Content here
</div>
<!-- /wp:namespace/block-name -->
```

- **Attributes:** Stored in the opening comment JSON.
- **Save Function:** Returns the inner HTML between delimiters.
- **Dynamic Rendering:** `render.php` generates the content dynamically; attributes are passed via PHP.

## 3. Registration of a Block

Blocks in WordPress are typically bundled in a plugin and registered on both the server and client-side using `block.json` metadata. While it's possible to register blocks solely on the client-side, best practices strongly advise registering them on both. This dual registration is crucial for enabling server-side features such as Dynamic Rendering, Block Supports, Block Hooks, and Style Variations.

### Registering a block with PHP (server-side)
Block registration on the server takes place in the main plugin PHP file, hooked on `init`.

#### Registering all blocks at once with a metadata collection (recommended)
Since WordPress 6.8, the recommended way to register blocks is `wp_register_block_types_from_metadata_collection()`. This function registers all block types defined in a `blocks-manifest.php` file in a single call.

```php
function my_plugin_register_blocks() {
    wp_register_block_types_from_metadata_collection(
        __DIR__ . '/build',
        __DIR__ . '/build/blocks-manifest.php'
    );
}
add_action( 'init', 'my_plugin_register_blocks' );
```

#### Registering a metadata collection separately (WordPress 6.7+)
If you need more control over how individual blocks are registered, you can use `wp_register_block_metadata_collection()` to register the manifest first, then call `register_block_type()` for each block individually.

```php
function my_plugin_register_blocks() {
    wp_register_block_metadata_collection(
        __DIR__ . '/build',
        __DIR__ . '/build/blocks-manifest.php'
    );

    register_block_type( __DIR__ . '/build/block-one' );
    register_block_type( __DIR__ . '/build/block-two', array(
        'render_callback' => 'my_plugin_render_block_two',
    ) );
}
add_action( 'init', 'my_plugin_register_blocks' );
```

#### Registering a single block with `register_block_type()`
The `register_block_type()` function registers a single block type by reading metadata from a `block.json` file. This remains a valid approach, particularly for plugins with a single block or when targeting WordPress versions older than 6.7.

```php
function minimal_block_register_block() {
    register_block_type( __DIR__ . '/build' );
}
add_action( 'init', 'minimal_block_register_block' );
```

#### PHP-only blocks with auto-registration
For blocks that only need server-side rendering, you can register them exclusively in PHP using the `autoRegister` flag.

```php
register_block_type( 'my-plugin/server-block', array(
    'render_callback' => function( $attributes ) {
        $wrapper_attributes = get_block_wrapper_attributes();
        return sprintf( '<div %1$s>Server content</div>', $wrapper_attributes );
    },
    'supports' => array(
        'autoRegister' => true,
        'color' => array( 'background' => true ),
    ),
) );
```

### Registering a block with JavaScript (client-side)
When the block has already been registered on the server, you only need to register the client-side settings in JavaScript using the `registerBlockType` method from the `@wordpress/blocks` package.

```javascript
import { registerBlockType } from '@wordpress/blocks';
import { useBlockProps } from '@wordpress/block-editor';
import metadata from './block.json';

const Edit = () => <p { ...useBlockProps() }>Hello World - Block Editor</p>;
const save = () => <p { ...useBlockProps.save() }>Hello World - Frontend</p>;

registerBlockType( metadata.name, {
    edit: Edit,
    save,
} );
```

## 4. The Block Wrapper

### Core Purpose
Every block is contained within an HTML wrapper that must include specific attributes to function correctly in the editor and on the front end. These attributes handle styling, accessibility, block supports, and identification.

### Wrapper Element Requirements
- Must be a standard DOM element (e.g., `<div>`, `<p>`) or a React component that spreads props to native elements.
- **Not supported:** React Fragments (`<Fragment>`) or `<ServerSideRender>` as wrappers.

### Hook & Function Mapping

| Context | Tool | Purpose | Returns |
| :--- | :--- | :--- | :--- |
| **Editor (JS)** | `useBlockProps()` | Defines wrapper attributes in the Edit component. | Object of props for JSX spread (`{ ...useBlockProps() }`). |
| **Save (JS)** | `useBlockProps.save()` | Defines wrapper attributes for the saved database markup. | Object of props for JSX spread (`{ ...useBlockProps.save() }`). |
| **Dynamic Render (PHP)** | `get_block_wrapper_attributes()` | Outputs wrapper attributes in PHP render logic. | HTML string of attributes (e.g., `class="wp-block-..."`). |

### Automatic Attributes Included
When using these tools, the following are automatically applied:
- **Block Class:** `wp-block-{namespace}-{name}` for targeted styling.
- **Supports Classes/Styles:** Generated from `block.json` supports (e.g., color, spacing, alignment).
- **Accessibility:** `tabindex`, `role="document"`, `aria-label`.
- **Identification:** Unique `id`, `data-block`, `data-type`, `data-title` attributes.

### Implementation Examples

**JavaScript: Edit & Save Components**
```javascript
import { useBlockProps } from '@wordpress/block-editor';

// Edit Component (Editor View)
const Edit = () => (
    <p { ...useBlockProps() }>Hello World - Block Editor</p>
);

// Save Function (Database/Frontend View)
const save = () => (
    <p { ...useBlockProps.save() }>Hello World - Frontend</p>
);

registerBlockType('namespace/block-name', {
    edit: Edit,
    save,
});
```

**PHP: Dynamic Render (`render.php`)**
```php
<div <?php echo get_block_wrapper_attributes(); ?>>
    <?php esc_html_e( 'Dynamic Block Content', 'text-domain' ); ?>
</div>
```

### Adding Custom Attributes/Classes
Pass additional props directly to the hook/function:

**JS:**
```javascript
const blockProps = useBlockProps({
    className: 'my-custom-class',
    'data-custom': 'value',
});
return <div { ...blockProps }>Content</div>;
```

**PHP:**
```php
$wrapper_attrs = get_block_wrapper_attributes([
    'class' => 'my-custom-class',
    'data-custom' => 'value',
]);
echo "<div $wrapper_attrs>Content</div>";
```

## 5. The Editor & UI Components

### Core Purpose
The Block Editor is a React Single Page Application (SPA). Every block's editor view is rendered by the `edit` function defined in the block registration settings.

### Edit Component Props Contract

| Prop | Type | Description |
| :--- | :--- | :--- |
| `attributes` | Object | Key-value pairs of the block's current data. |
| `setAttributes` | Function | Updates the `attributes` object and triggers a re-render. |
| `isSelected` | Boolean | `true` if the block is currently selected in the editor. |
| `className` | String | CSS class applied to the wrapper element. |
| `clientId` | String | Unique identifier for this specific block instance. |

### Component Library Mapping

#### `@wordpress/components` (Generic UI)
Standard UI elements for forms, layouts, and interactions.

| Component | Usage |
| :--- | :--- |
| `TextControl` | Input fields for text/numbers. |
| `ToggleControl` | Boolean switches. |
| `Panel` / `PanelBody` | Collapsible containers for sidebar controls. |
| `ColorPalette` | Color selection inputs. |
| `ExternalLink` | Links to external resources. |

#### `@wordpress/block-editor` (Block-Specific)
Components and hooks specifically designed for block logic and editor integration.

| Component/Hook | Usage |
| :--- | :--- |
| `RichText` | Editable text container that supports formatting/HTML. |
| `InnerBlocks` | Container for nested/block inheritance. |
| `BlockControls` | Wrapper for the top Block Toolbar (visible when selected). |
| `InspectorControls` | Wrapper for the Settings Sidebar (visible when selected). |

### UI Control Patterns

#### 1. Block Toolbar (`<BlockControls>`)
- **Visibility**: Only appears when the block is **selected** and in **visual mode**.
- **Purpose**: Inline editing actions or alignment controls specific to the selection.
- **Constraint**: Do not use for global block settings (e.g., width, color). Use the Sidebar for those.

#### 2. Settings Sidebar (`<InspectorControls>`)
- **Visibility**: Appears in the right-hand sidebar when the block is **selected**.
- **Purpose**: Block-level configuration requiring more screen space or complex inputs.
- **Constraint**: Controls here remain visible even in HTML editing mode.

### Canonical Implementation Snippet

```javascript
import { useBlockProps } from '@wordpress/block-editor';
import { BlockControls, InspectorControls, RichText, TextControl } from '@wordpress/components';
import { ToolbarGroup, AlignmentToolbar, PanelBody, ColorPalette } from '@wordpress/components';

export default function Edit({ attributes, setAttributes }) {
    const { content, alignment, textColor } = attributes;

    return (
        <>
            {/* Block Toolbar: Inline Controls */}
            <BlockControls>
                <ToolbarGroup>
                    <AlignmentToolbar
                        value={alignment}
                        onChange={(newAlign) => setAttributes({ alignment: newAlign || 'none' })}
                    />
                </ToolbarGroup>
            </BlockControls>

            {/* Settings Sidebar: Block Configuration */}
            <InspectorControls>
                <PanelBody title="Settings">
                    <TextControl
                        label="Message"
                        value={content}
                        onChange={(val) => setAttributes({ content: val })}
                    />
                    <ColorPalette
                        value={textColor}
                        onChange={(newColor) => setAttributes({ textColor: newColor })}
                    />
                </PanelBody>
            </InspectorControls>

            {/* Block Wrapper */}
            <p { ...useBlockProps({ className: attributes.className }) }>
                <RichText
                    tagName="span"
                    value={content}
                    onChange={(newContent) => setAttributes({ content: newContent })}
                    style={{ color: textColor, textAlign: alignment }}
                />
            </p>
        </>
    );
}
```

## 6. Markup Representation & Storage

### Core Syntax Rule
Blocks are stored in the database or templates using **HTML comment delimiters**. This ensures block markup remains technically valid HTML.

### Naming Conventions

| Type | Prefix | Format | Example |
| :--- | :--- | :--- | :--- |
| **Core Block** | `wp:` | `wp:block-name` | `<!-- wp:image -->` |
| **Custom Block** | `wp:` | `wp:namespace/name` | `<!-- wp:plugin/block -->` |

### Storage Patterns

#### 1. Static Blocks (Standard)
The `save` function generates the HTML stored in the database, wrapped in delimiters.

```html
<!-- wp:namespace/block-name {"attributes":{"key":"value"}} -->
<div class="wp-block-namespace-block-name">
    Generated HTML content here
</div>
<!-- /wp:namespace/block-name -->
```

#### 2. Dynamic Blocks (`save = null`)
Only the delimiter and attributes are stored. The front-end HTML is generated server-side via `render.php`.

```html
<!-- wp:namespace/block-name {"attributes":{"key":"value"}} /-->
```

### Validation Contract
The Block Editor validates that the markup stored in the database matches the output of the `save` function.
- **Match**: Content renders correctly in both Editor and Frontend.
- **Mismatch**: Triggers a "Validation Error" (block appears as a code block or warning).
- **Resolution**: Implement **Block Deprecations** to handle changes to the `save` function output over time.

## 7. Static vs. Dynamic Rendering

### Core Purpose
Blocks must define how their front-end markup is generated. This decision determines whether content is stored as fixed HTML in the database (Static) or generated on-the-fly by PHP when the page loads (Dynamic).

### Comparison Matrix

| Feature | Static Rendering | Dynamic Rendering |
| :--- | :--- | :--- |
| **Storage** | Full HTML markup + attributes. | Attributes only (or minimal fallback HTML). |
| **Generation** | Generated once in `save()` function. | Generated on every request via PHP (`render.php`). |
| **Performance** | Fast (served directly from DB); no PHP execution. | Slower (requires PHP execution per request). |
| **Updates** | Requires re-saving the block to change markup. | Changes apply immediately across all instances. |
| **Use Case** | Fixed layouts, Button, Spacer, Separator. | Latest Posts, Dynamic Galleries, User-specific data. |

### Implementation Paths

#### 1. Static Rendering (Standard)
Defined entirely in JavaScript. The `save` function returns JSX/HTML.

**`block.json`:** No specific property needed (defaults to static).
**`save.js`:**
```javascript
const save = () => {
    return <p className="my-static-block">Static Content</p>;
};
```

#### 2. Dynamic Rendering (PHP)
Defined via a PHP file that generates the output.

**`block.json`:**
```json
{
    "render": "file:./render.php"
}
```
**`render.php`:**
```php
<div class="my-dynamic-block">
    <?php echo esc_html( 'Dynamic Content' ); ?>
</div>
```

#### 3. PHP-Only Registration (Legacy/Script-less)
For blocks without a JS build step, defined via `register_block_type()` in PHP.

**PHP:**
```php
register_block_type( 'namespace/block-name', [
    'render_callback' => 'my_dynamic_render_callback',
] );
```

### Rendering Strategies

#### Pure Static
- **Save:** Returns HTML.
- **Render:** Not defined.
- **Result:** Database contains the exact HTML displayed on the front end.

#### Pure Dynamic
- **Save:** Returns `null`.
- **Render:** Defined in PHP.
- **Result:** Database contains only delimiters and attributes. The `render.php` generates all HTML.

#### Hybrid (Fallback/Modification)
- **Save:** Returns HTML.
- **Render:** Defined in PHP.
- **Result:** 
    - **Active:** PHP renders the block, optionally modifying the saved HTML via `$content`.
    - **Inactive:** The static HTML from `save()` is displayed as a fallback if the plugin is disabled.

**Note:** When using `InnerBlocks` in a dynamic block, you must include `<InnerBlocks.Content />` in your `save()` function to preserve nested blocks in the database.

## 8. JavaScript Execution Contexts

### Core Purpose
JavaScript in block development must be transformed into browser-compatible code. The execution context determines how assets are loaded and managed.

### Execution Contexts

#### 1. Build Process (Standard)
Uses `@wordpress/scripts` (`wp-scripts`) to bundle ESNext/JSX into browser-compatible output.

| Mode | Command | Behavior |
| :--- | :--- | :--- |
| **Development** | `npm start` | Watch mode; skips minification; generates source maps. |
| **Production** | `npm run build` | Compiles, bundles, and minifies assets for deployment. |

**Configuration:** Pre-configured out-of-the-box. Customization is possible via a local `webpack.config.js`.

#### 2. No Build Process (Script-less)
Directly interacts with the global `wp.*` object provided by WordPress. Requires manual dependency management.

- **Syntax**: Standard ES5/ES6 JavaScript (no JSX/Babel).
- **Dependencies**: Manually define required packages in the `deps` array of `wp_enqueue_script()`.
- **Loading Hook**: Use `enqueue_block_editor_assets` for editor scripts.

### Script Enqueuing Checklist

#### Build Process Setup
1.  Initialize with `npm init @wordpress/block my-plugin`.
2.  Write source code in `src/` (ESNext/JSX).
3.  Reference assets in `block.json` using `file:` prefix:
    -   `"editorScript": "file:./build/index.js"`
    -   `"style": "file:./build/style.css"`

#### No-Build Enqueuing Example
**PHP (`plugin.php`):**
```php
function example_enqueue_block_variations() {
    wp_enqueue_script(
        'example-variations',
        get_template_directory_uri() . '/assets/js/variations.js',
        array( 'wp-blocks' ), // Manual dependency declaration
        wp_get_theme()->get( 'Version' ),
        false
    );
}
add_action( 'enqueue_block_editor_assets', 'example_enqueue_block_variations' );
```

**JS (`variations.js`):**
```javascript
wp.blocks.registerBlockVariation(
    'core/media-text',
    {
        name: 'media-text-custom',
        title: 'Media & Text Custom',
        attributes: { align: 'wide', backgroundColor: 'tertiary' },
    }
);
```
